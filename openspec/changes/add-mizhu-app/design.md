# 迷住团购管家 — 设计文档

## 1. 业务模型

### 1.1 返现体系全景

```
┌─────────────────────────────────────────────────────────────────────┐
│                        一笔订单的返现计算管道                          │
│                                                                     │
│  Layer 0: 实付金额 (基础)                                           │
│     ├──▶ 大套购 ── 活动级总额分档，档位拆分不共享                     │
│     │      3000→100, 5000→200, 10000→500                           │
│     │      → 按各商品实付比例分摊到每件商品                            │
│     │                                                               │
│     ├──▶ 平台保价退款 ──┐ (店铺侧，收货后即时)                       │
│     ├──▶ 店铺保价退款 ──┘ (店铺侧，收货+14天)                        │
│     │                                                               │
│     ├──▶ 单品返现条目 ──┤ (店铺侧，收货+14天)                        │
│     │    下单即返 / 小红书晒单 / 组合购 / 其他自定义                   │
│     │                                                               │
│     ├──▶ 品牌套购 ──────┘ (店铺侧，收货+14天)                        │
│     │    基准 = 下单价 - 平台保价 - 店铺保价 - 单品返现合计            │
│     │    返现 = 基准 × (1 - 系数)                                    │
│     │                                                               │
│     ├──▶ 迷住返(加返) ── (迷住侧，收货+14天)                         │
│     ├──▶ 舰长/装修神券 ─ (迷住侧，收货+14天，互斥)                   │
│     ├──▶ 大套购分摊 ──── (迷住侧，手动结算)                          │
│     │                                                               │
│     └──▶ 迷住保价 ────── (迷住侧，收货+14天)                         │
│          一类: 基于 下单价-单品返现-平台保价-店铺保价 的到手价比对     │
│          二类: 基于除神券外所有返现项的到手价比对                      │
│          用户输入迷住保价到手价 → 自动计算差价                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 付款方分类

所有返现按付款方分为两侧：

- **店铺侧**：平台保价、店铺保价、下单即返、小红书晒单、组合购、其他自定义、品牌套购
- **迷住侧**：迷住返(加返)、舰长神券、装修神券、大套购分摊、迷住保价

收款方永远是使用者自己，追踪两路钱分别到没到。

## 2. 数据模型

### 2.1 实体关系

```
Activity (活动)
  ├── bigPurchaseRules [档位列表]
  ├── SettlementBatch × N (大套购结算批次)
  │     ├── id, date
  │     ├── totalAmount (本批总额)
  │     ├── totalCashback (本批返现)
  │     └── allocations: [{ itemId, amount }, ...]
  │
  └── Item (商品) × N
        ├── 基本信息 (name/channel/orderNo/price/gift/remark)
        ├── confirmDate (确认收货日期)
        ├── status (待收货/待返现/已完成/已取消)
        ├── brandCoefficient (品牌套购系数, null=不参与)
        ├── 迷住保价 (Item 上, 只有一个)
        │     ├── mizhuProtectType (一类/二类/null)
        │     └── mizhuProtectPrice (迷住保价到手价)
        ├── CashbackEntry × N (返现条目)
        │     ├── type (12种枚举, →自动映射 payer)
        │     ├── customLabel (type='other' 时用)
        │     ├── amount (手填或自动算或预估)
        │     ├── dueDate (应到日期, null=大套购手动结算)
        │     └── isEstimated (true=预估中, false=已固定)
        │     ◇ 无收款状态字段, 收款由 Item + PaymentRecord 管理
        └── PaymentRecord × N (收款记录)
              ├── date, amount, payer (店铺/迷住)
              └── 可删除
```

### 2.2 CashbackEntry 分类

返现条目按 amount 来源分为三类：

**手动固定型（9种）：**
用户输入金额后固定不变。包括：平台保价退款、店铺保价退款、下单即返、小红书晒单、组合购、其他(自定义)、迷住返(加返)、舰长神券、装修神券。

**自动计算型（2种）：**
amount 由计算引擎推导，每次 Item 更新后重算。

- 品牌套购（brand）：brandCoefficient 填了就生成 Entry，基准依赖其他 Entry 金额
- 迷住保价（mizhu_protect）：保价返现 > 0 才生成 Entry

**预估→固定型（1种）：**
- 大套购分摊（big_purchase）：预估 amount = price × 0.05，结算后更新为实际值

### 2.3 CashbackEntry 类型枚举

| type | 含义 | payer | 到账规则 | amount 来源 |
|------|------|-------|---------|------------|
| platform_refund | 平台保价退款 | 店铺 | 收货后即时 | 手填(支持多次累计) |
| store_refund | 店铺保价退款 | 店铺 | 收货+14天 | 手填(支持多次累计) |
| instant_cashback | 下单即返 | 店铺 | 收货+14天 | 手填 |
| xhs_post | 小红书晒单 | 店铺 | 收货+14天 | 手填 |
| combo | 组合购 | 店铺 | 收货+14天 | 手填 |
| other | 其他(自定义) | 店铺 | 收货+14天 | 手填(需customLabel) |
| brand | 品牌套购 | 店铺 | 收货+14天 | 自动计算 |
| mizhu_extra | 迷住返(加返) | 迷住 | 收货+14天 | 手填 |
| captain_coupon | 舰长神券 | 迷住 | 收货+14天 | 手填 |
| renovation_coupon | 装修神券 | 迷住 | 收货+14天 | 手填 |
| mizhu_protect | 迷住保价 | 迷住 | 收货+14天 | 自动计算 |
| big_purchase | 大套购分摊 | 迷住 | 手动结算 | 预估→固定 |

### 2.4 动态计算（每次 Item 更新后重算）

```
Step 1: 品牌套购 Entry (type='brand')
  如果 brandCoefficient != null:
    基准 = price - 平台保价 - 店铺保价 - Σ(店铺侧手填条目)
    Entry.amount = max(0, 基准 × (1 - brandCoefficient))
  如果 brandCoefficient == null:
    删除该 Entry

Step 2: 迷住保价 Entry (type='mizhu_protect')
  如果 mizhuProtectType != null && mizhuProtectPrice != null:
    差价 = 原到手价 - 迷住保价到手价
    差价 > 0 → Entry.amount = 差价
    差价 > 0 → Entry.amount = 差价
    差价 ≤ 0 → 删除该 Entry
  如果 mizhuProtectType == null:
    删除该 Entry

Step 3: 大套购 Entry (type='big_purchase')
  如果 isEstimated == true:
    Entry.amount = price × 0.05 (固定常量)
  如果 isEstimated == false:
    跳过 (已固定)
```

### 2.5 关键聚合字段（实时计算，不单独存储）

- **店铺侧合计** = Σ(Entry where payer='店铺')
- **迷住侧合计** = Σ(Entry where payer='迷住')
- **到手价** = price - 店铺侧合计 - 迷住侧合计
- **已返合计** = Σ(PaymentRecord)
- **店铺已返** = Σ(PaymentRecord where payer='店铺')
- **迷住已返** = Σ(PaymentRecord where payer='迷住')
- **店铺待返** = 店铺侧合计 - 店铺已返
- **迷住待返** = 迷住侧合计 - 迷住已返
- **待返总额** = 店铺待返 + 迷住待返

### 2.6 大套购计算

**预估阶段（创建商品时自动生成）：**
- 预估分摊 = price × 0.05（固定常量）
- 自动生成 CashbackEntry { type:'big_purchase', amount:预估值, isEstimated:true, dueDate:null }
- 预估完全独立，不依赖其他订单状态

**分档拆分算法（档位不共享）：**
```
金额按从高到低的档位依次消耗

例: ¥13,500，档位 [10000→500, 5000→200, 3000→100]
→ 10000×1 (剩3500) → 3000×1 (剩500)
→ 返现 = 500 + 100 = ¥600
```

**分摊算法：**
- 按各商品实付金额占本批总额的比例分摊

**结算时：**
- Entry.amount 更新为实际分摊值
- Entry.isEstimated → false
- 自动生成 PaymentRecord(payer='迷住')

**撤销结算时：**
- Entry.amount 回退为 price × 0.05
- Entry.isEstimated → true
- 对应 PaymentRecord 删除

## 3. 状态流转

```
                    创建商品
                       │
                       ▼
                  ┌─────────┐
                  │ 待收货   │ ← 默认状态
                  └────┬────┘
                       │ 确认收货 (单个/批量)
                       │ → 启动14天倒计时
                       │ → 平台保价即时到账
                       ▼
                  ┌──────────┐
        ┌────────│ 待返现    │────────┐
        │         └────┬─────┘        │
        │              │              │
   重置状态       待返归零         取消订单
  (删收款记录     (自动)        (前置条件:
   回待收货)                  无返现记录
        │              │         且未结算)
        │              ▼          │
        │         ┌─────────┐     │
        └────────│ 已完成   │     │
                  └─────────┘     │
                                 ▼
                            ┌─────────┐
                  重置状态   │ 已取消   │
                  ─────────▶ └─────────┘
```

### 状态切换动作

| 动作 | 效果 | 前置条件 |
|------|------|---------|
| 确认收货 | 待收货→待返现，启动14天倒计时 | 当前为待收货 |
| 收到返现 | 新增收款记录，更新待返，归零则自动已完成 | 当前为待返现或已完成 |
| 取消订单 | →已取消 | 无返现记录且未参与大套购结算 |
| 重置状态 | →待收货，删除所有收款记录 | 非待收货状态 |
| 删除 | 硬删除商品及所有关联数据 | 任何时候 |

## 4. 保价计算

### 4.1 平台/店铺保价

用户直接输入补偿金额，作为返现条目。

### 4.2 迷住保价

用户输入迷住保价到手价 + 类型（一类/二类），app 自动计算。

```
一类保价:
  原一类到手价 = 原下单价 - 单品返现 - 平台保价 - 店铺保价
  保价返现 = 原一类到手价 - 迷住保价到手价

二类保价:
  原二类到手价 = 原下单价 - 单品返现 - 店铺活动 - 品牌套购
                - 大套购 - 平台保价 - 店铺保价
  保价返现 = 原二类到手价 - 迷住保价到手价

规则:
  保价返现 > 0 → 生成返现条目
  保价返现 ≤ 0 → 记录迷住保价到手价但不生成返现（保价到手价不优于原到手价）
```

注意：迷住保价保的是到手价，用户输入的就是迷住承诺的到手价，直接与原到手价比差，不需要重算扣减项。

## 5. 边缘场景规则

| 场景 | 规则 |
|------|------|
| 返现总额 > 实付金额 | 异常，不允许保存 |
| 到手价 < 0 | 异常，不允许保存 |
| 保价好价 ≥ 下单价 | 不允许记录 |
| 保价返现 ≤ 0 | 记录迷住保价到手价但不产生返现 |
| 已参与大套购结算的商品 | 下单价锁定，不允许修改 |
| 重置状态 | 删除收款记录，回待收货，其他不变 |
| 删返现记录导致超期 | 待返页面正常显示超期 |
| 恢复已取消订单 | 累入大套购未结算金额，已结算不受影响 |
| 大套购撤销 | 只支持整批撤销，不支持单件 |
| 删除商品 | 硬删除，连级删除相关条目 |
| 取消订单 | 软删除（可恢复） |
| 金额精度 | 保留 2 位小数，计算用整数分（×100） |

## 7. 技术架构

```
┌──────────────────────────────────────────────┐
│                  PWA App                     │
├──────────────────────────────────────────────┤
│  UI Layer (Vue 3 + Vant + Tailwind CSS)      │
│    首页 / 商品清单 / 待返清单 / 结算 ...        │
├──────────────────────────────────────────────┤
│  Calculation Engine (TypeScript)             │
│    品牌套购 / 大套购分档分摊 / 保价差价 /       │
│    到账日期推算 / 状态流转                      │
├──────────────────────────────────────────────┤
│  Data Layer (Dexie.js / IndexedDB)           │
│    活动 / 商品 / 返现条目 / 结算批次 /          │
│    收款记录                                    │
├──────────────────────────────────────────────┤
│  PWA Shell                                   │
│    Service Worker (离线缓存)                  │
│    Web Manifest (安装到桌面)                   │
└──────────────────────────────────────────────┘
```

### 技术选型理由

- **Vue 3 + TypeScript**：类型安全（返现计算逻辑复杂），模板语法对新手友好
- **Vite**：极快的开发热更新
- **Vant**：Vue 生态最成熟的移动端组件库（折叠面板/弹窗/下拉/日期选择）
- **Tailwind CSS**：自定义样式
- **Dexie.js**：IndexedDB 封装，支持复杂查询、索引、事务
- **vite-plugin-pwa**：零配置离线支持
- **ECharts**：统计图表
